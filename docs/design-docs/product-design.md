## 产品定位

MealMate Lite 是一个**以 AI 对话为核心交互**的家庭饮食规划 App。

用户通过自然语言对话完成所有操作——录入菜品、生成计划、管理库存、查看清单。不需要填表单、不需要学操作，说一句话就行。

### 目标用户

- 私人家庭（1-6 人），非公开服务
- 日常做饭的家庭成员（负责"今天吃什么"的那个人）
- 不想每天花脑力想菜、不想忘记买菜、不想浪费食材的人

### 核心价值

解决"每天吃什么"这个高频低价值决策，把它交给 AI：

| 痛点 | MealMate 的回答 |
|---|---|
| 每天想吃什么太累 | AI 根据口味和历史自动生成周计划 |
| 菜品信息散落在脑子里 | 对话录入，渐进式完善，形成家庭菜品库 |
| 买菜漏买 / 重复买 | AI 对比计划和库存，自动列采购清单 |
| 食材放过期浪费 | 追踪库存过期，AI 优先安排临期食材 |
| 做多了剩饭不知道怎么消耗 | 记录剩菜，下轮计划优先消耗 |
| 家人口味不同难兼顾 | 多角色偏好，AI 计划兼顾所有人 |

### 与通用 ChatGPT 的区别

| ChatGPT | MealMate |
|---|---|
| 不记得你家有什么菜 | 持久化菜品库，越用越丰富 |
| 不知道冰箱里有什么 | 追踪库存，实时扣减 |
| 每次对话从头来 | 多轮上下文持续 + 历史偏好注入 |
| 输出纯文字无法操作 | 回复中嵌入可交互组件（勾选、卡片、日历） |
| 无提醒能力 | 备菜/过期/采购提醒推送到手机 |
| 无法多端共享 | 家庭多设备同步同一份数据 |

## 功能全景

### 1. AI 对话（核心入口）

对话页是 App 的首页和主要操作入口。用户日常通过对话完成大部分操作。

**能力**：
- 自然语言驱动所有业务操作（增删改查菜品、生成计划、管理库存、出清单…）
- 多轮上下文连续，AI 记住前面说过什么
- 单轮可触发多个动作（"加三道菜然后出个计划"）
- 流式逐字回复，响应感知快
- 批量菜谱和覆盖周计划时显示固定的结构化确认面板；这是 v0.1 对纯文本流的唯一例外，不提供通用卡片/混合消息渲染
- AI 回复中嵌入可交互组件（菜谱卡片、可勾选清单、日历卡片）
- 快捷操作按钮（"换一批""少辣""只要素的"）
- 断网发送 AI 消息立即提示失败，保留草稿并提供手动“重新发送”；不会在后台自动补发
- v0.1 中，每台设备只维护自己的对话上下文与模型选择；家庭菜谱和计划仍共享

**示例对话**：
- "加个红烧肉" → 创建菜品
- "红烧肉的食材帮我补一下" → 多轮补充完善
- "帮我生成 20 个家常川菜" → 生成预览，用户在固定面板确认后批量创建菜品库
- "安排下周的菜，少辣" → 生成 7 天 × 3 餐计划
- "周三中午换一个" → 调整单餐
- "冰箱里有鸡蛋牛奶西红柿猪肉" → 批量建库存
- "出采购清单" → 基于计划和库存差值生成
- "有什么快过期的" → 推荐消耗临期食材的菜
- "明早提醒我泡黄豆" → 设置提醒

### 2. 菜品管理

家庭私有菜品库，AI 辅助渐进式完善。

**能力**：
- 最小录入：只需一个菜名即可建档
- 渐进式完善：后续对话中逐步补充食材、步骤、标签、图片、备注
- 批量冷启动：一句话让 AI 生成 20-50 道菜快速填充菜品库
- 标签体系：菜系、口味、适合人群、烹饪难度、耗时等
- 搜索与筛选：按名称、标签、食材搜索
- 家庭拿手菜标记：区分"咱家会做的"和"AI 推荐的新菜"
- 首次尝试日期记录

> v0.1 搜索仅做名称、标签和自由文本食材的不区分大小写包含匹配；菜谱允许重名。

### 3. 周计划

AI 生成一周三餐安排，支持对话式调整。

**能力**：
- AI 自动生成 7 天 × 3 餐（早/午/晚）计划
- 考虑因素：家庭口味偏好、忌口、近期不重复、食材复用、临期消耗
- 一餐多菜：荤素搭配、干稀搭配
- 对话调整："周三中午换一个""再加个汤""这周多安排点鱼"
- 日历视图展示

> v0.1 固定使用 `Asia/Shanghai`，一餐一菜；生成时优先避开过去 7 天已安排的菜谱，菜谱不足时允许复用并明确说明。

### 4. 食材与库存

追踪家里有什么、有多少、什么时候过期。

**能力**：
- 结构化食材字典：名称、类别（蔬菜/肉类/调料…）、默认单位、保质期
- 同义词合并："西红柿"="番茄"
- 库存记录：数量、单位、过期日期
- 对话操作库存："买了鸡蛋 30 个""用掉了三个"
- 批量初始化："冰箱里有…"一句话建多条库存（目标 30 秒内）
- 过期预警：分类浏览时标记临期食材
- 采购自动入库：采购清单完成后自动增加库存
- 计划自动扣减：做完菜后自动减少对应食材库存

### 5. 采购清单

基于计划所需 - 现有库存 = 需要买的东西。

**能力**：
- AI 自动生成：对比周计划食材需求和当前库存，输出差值清单
- 按食材类别分组展示
- 勾选确认：买完的项目打勾
- 完成入库：勾选完成后相关食材自动进入库存
- "差什么"检查：对比今日菜品 × 库存，一键补充采购清单

### 6. 剩菜与食材复用

减少浪费的智能逻辑。

**能力**：
- 剩菜记录：做完饭后 AI 追问"有剩吗？"，记入剩菜池
- 优先消耗：下轮生成计划时，优先安排消耗剩菜和临期食材
- 食材复用约束：AI 生成计划时优先组合共享食材的菜品（周一用西红柿做番茄炒蛋，周三做番茄牛腩），减少采购品类

### 7. 家庭角色与偏好

从一段文本升级为多角色口味管理。

**能力**：
- 基础偏好：一段可编辑文本，注入 AI 的 System Prompt
- 家庭角色：爸爸/妈妈/孩子/老人…各自口味和忌口
- 角色切换：对话时选择"当前谁在说话"，AI 动态调整建议
- AI 计划兼顾：生成周计划时平衡多角色需求

### 8. 提醒通知

主动推送关键时间节点。

**类型**：

| 提醒 | 触发 | 示例 |
|---|---|---|
| 备菜提醒 | 计划餐次前 N 小时 | "明天中午做糖醋排骨，今晚记得腌肉" |
| 过期提醒 | 食材过期前 N 天 | "牛奶后天过期" |
| 计划提醒 | 每周固定时间，计划未生成 | "这周还没安排菜单" |
| 采购提醒 | 清单生成后 / 指定日期 | "周末该去买菜了" |
| 自定义提醒 | 用户对话设置 | "明早提醒我泡黄豆" |

**管理**：设置页可查看所有提醒规则，按类型开关。

### 9. 统计与回顾

帮助感知饮食习惯，不做精确营养分析。

**能力**：
- 每周回顾卡片："本周 12 道菜，最常用食材猪肉，尝试了 2 道新菜"
- 频次统计：最常做的菜 Top N、最常用食材 Top N
- 品类分布：荤素比、菜系分布饼图
- 时间趋势：月度变化折线

### 10. 视觉识别

拍照降低操作成本。

**场景**：
- 拍照录菜：拍一道做好的菜 → 识别菜名 → 录入菜品
- 拍照盘库存：拍冰箱/菜篮 → 识别食材列表 → 确认后批量入库
- 与文字互补：文字适合初始化和精确操作，拍照适合日常快速更新

### 11. 智能推荐

系统主动帮助，不只被动响应。

**能力**：
- 口味学习：从历史计划和调整行为推断偏好变化
- 季节推荐：换季时主动推荐应季菜品
- 复购预测：追踪消耗速度 → 预测补货时机 → 主动提醒或加入采购清单
- 浪费追踪：记录过期丢弃量 → 计算浪费金额 → AI 调整采购量建议

### 12. 语音交互

解放双手，做菜场景可用。

**场景**：
- 做菜时语音操作："鸡蛋用完了" → 扣库存
- 逐步骤语音烹饪引导："下一步""换个做法""少放多少盐"
- 实时替代问答："没有料酒用什么代替"

### 13. 多端与共享

家庭多设备同步同一份数据。

**能力**：
- 单家庭单部署：一个后端实例只服务一个家庭。首台设备用部署者线下提供的 bootstrap secret 初始化；家庭码仅用于后续设备加入，无需注册账号
- 多设备共享：家人各自手机看到同一份菜品库、计划和库存；对话历史和所选模型仅属于各自设备
- 所有已加入设备权限相同：可编辑家庭偏好、轮换家庭码，以及在设置中查看和撤销指定设备；轮换后旧家庭码立即失效，既有设备继续可用
- 设备令牌不按时间过期，仅在注销、被撤销或部署者恢复重置时失效；恢复重置不删业务数据，但会撤销全部旧设备令牌
- 离线可用：断网时可浏览所有本地数据；仅菜品名称/标签/删除等轻操作自动同步。家庭偏好只存本地草稿、联网后由用户手动保存；AI 消息保留草稿并由用户手动重发
- 多家庭共享（v1.3 重新设计）：引入明确的家庭实体、数据隔离与加入流程；v0.1 家庭码只加入当前唯一家庭，不能创建第二个家庭

## 关键行为规则

### 对话管理

- 每台设备保持一个会话，不支持多会话或新建对话；设备间不共享上下文
- 同一设备同一时间只处理一条 AI 消息；请求进行中时输入框保留但发送按钮禁用。服务端也会拒绝绕过界面的第二个请求，v0.1 不排队、不提供取消。中断后用户可重试原消息；若改为发送新消息，旧的可恢复请求即失效，不能稍后插入会话
- 上下文窗口为最近 20 个完整 user/assistant 轮次，超出部分会从服务端和该设备永久删除；对应聊天回执同步清除正文、工具参数和结果，只保留不可重执行的幂等墓碑，旧请求再次重试会提示已过期
- 首次注册完成且无历史消息时，由 App 本地显示固定的 AI 风格欢迎文案 + 快捷操作气泡（"生成 20 个家常菜""安排下周的菜""加个红烧肉"）；该空状态不请求模型、不写入 Conversation，也不计入 20 轮

### 菜品管理

- 菜谱允许重名。按名称更新、删除或替换前必须先搜索；若没有或有多个候选，AI 要求用户澄清
- 删除支持两种方式：对话删除（"把红烧肉删了"）+ 菜品库滑动/长按删除；若当前或未来计划引用该菜谱，则拒绝删除并提示先替换或移除相关餐次
- 已删除菜谱不能靠普通编辑复活，只能在对话中明确“恢复某菜”；只搜索已删除菜谱，多个候选时要求选择
- 批量生成菜品后，App 用固定只读确认面板展示服务端预览列表，只提供“确认”操作，不允许在面板内删项或改名。默认跳过同名项并提示；用户要删项或改名必须继续对话，由模型提交完整新列表并取得新预览。新预览立即使该设备上一份同类预览失效；确认仅限发起预览的设备且 10 分钟内有效
- 菜品详情页可就地编辑名称和标签（轻操作）；复杂修改（食材、步骤）引导到对话
- MVP 不做图片上传和展示，字段保留

### 周计划

- 每周只有一份计划。再次生成同一周的计划时，AI 先展示草稿并提示“本周已有计划，要覆盖吗？”，仅发起预览的设备可在 10 分钟内确认覆盖；新预览会使该设备上一份同类预览失效。若预览后该周计划已被更新，旧确认会被拒绝，用户须重新生成预览；同周不保留版本历史
- 历史计划永久保留，周计划页默认展示当前周，可切换查看历史。每个餐次创建时保存菜谱名称快照，因此即便历史菜谱已删除也能正常展示
- 周计划页菜品卡片有"换一个"按钮，点击跳转对话页并预填消息

### 错误处理

- 仅无副作用的读取或尚未产生副作用的模型请求可自动重试；写入型 FC 不盲目重试，依靠幂等键避免重复执行
- SSE 60 秒没有事件或总时长达到 5 分钟时，以可重试错误结束；模型不可用时不自动切换到其它模型
- SSE 建立前失败显示普通错误；建立后以流内错误事件结束。未收到 `done` 的 assistant 增量只作为当前 `chatRequestId` 的内存草稿并标记失败，不进入正式历史；点击“重试”时先清除/替换这段不完整回复，再复用同一聊天请求 ID，避免重复文本。已完成则重放结果，仍在活动处理则提示稍后重试，服务端租约已过期则由本次用户操作恢复并复用已完成工具结果；系统不在后台自动恢复。同设备另一个请求仍在处理时显示“上一条消息仍在处理中”，不自动重试。离线从未送达的草稿点击“重新发送”才创建新聊天请求
- 确认面板 token 只在当前 App 进程内存保存；进程被杀或连接丢失后，用户重试同一聊天请求即可重放尚有效的面板。确认按钮直接调用确认 API，不再发送“确认”聊天消息，也不让模型执行提交
- 被服务端拒绝的离线轻操作先回滚到服务端返回的当前状态，再保留为“同步失败”记录，显示原因并可丢弃或基于当前数据重新编辑

### 首次体验

- 首次打开 App → 输入部署时提供的 bootstrap secret，由服务端完成一次性初始化并生成形如 `XXXX-XXXX-XXXX` 的 12 位家庭码 → 用户记下或分享给家人 → 注册完进入对话页 → App 显示本地欢迎空状态。家庭码输入忽略大小写、空格和连字符；后续设备仅能用家庭码注册。bootstrap secret 和家庭码从同一来源连续失败 5 次后，15 分钟内不可再尝试。
- 如果 bootstrap 已在服务端提交、但响应丢失导致首台设备没有收到令牌或家庭码，App 不伪装成未初始化重试；应提示联系部署者执行恢复重置，再重新加入。

## 交互原则

### 对话优先

- 对话页是首页和主入口，日常操作通过对话完成
- 其他页面（菜品库、周计划、库存、采购清单）是数据展示和浏览
- 重操作引导回对话，轻操作（勾选、改名、开关）就地完成

### 渐进式

- 菜品只需一个名字就能存在，细节后续慢慢补
- 库存一句话建立，精确数量以后再调
- 偏好从一段文字开始，后续升级为角色体系
- 不逼用户一次填完所有信息

### AI 主动性

- AI 不只被动回答，会主动追问和建议
- 做完饭追问"有剩吗？"
- 临期食材主动推荐"鸡蛋还有 3 天过期，要不要明天安排个蛋类菜？"
- 周日主动出回顾卡片
- 换季主动推荐应季菜

### 低门槛

- 30 秒建立首批库存
- 一句话批量生成菜品库冷启动
- 不需要学操作流程，会说话就会用

## 页面结构

| 页面 | 定位 | 核心功能 |
|---|---|---|
| 对话（首页） | 操作入口 | AI 多轮对话，所有操作从这里发起 |
| 菜品库 | 数据浏览 | 搜索、标签筛选、查看详情 |
| 周计划 | 数据浏览 | 日历视图展示本周三餐，一餐多菜卡片 |
| 库存 | 数据浏览 | 分类浏览、过期预警标记 |
| 采购清单 | 轻操作 | 按类别分组展示、勾选确认 |
| 统计 | 数据浏览 | 频次、分布、趋势、周回顾卡片 |

## AI 模型

- 用户可在设置页为当前设备切换 AI 模型；只显示部署者已启用、声明支持 Function Calling/SSE 且在发布前通过显式 verify 命令的模型
- 新设备首次进入对话页时自动选择部署端唯一的默认模型；之后只保留该设备自己的选择
- 切换立即生效，当前设备的下一轮对话使用新模型，不影响其它设备
- 模型不可用时明确提示失败，允许用户重试或手动换模型；不自动降级到其它提供方
- API Key 仅由部署者在受限服务器环境配置，不进入数据库、客户端、接口响应或日志
- 聊天请求的模型标识必须再次通过服务端 allowlist 校验；未知、停用或不兼容模型提示不可用
